# v2.0.4 - 監視付き Worker シャットダウン

> 開発プレリリースです。本リリースは `dev` から公開され、短期間で変更が
> 加えられる可能性があります。また、既定の `latest` イメージでは選択されません。

## 概要

v2.0.4 では、バックエンドの worker manager が所有するすべての worker に、共通の
スーパーバイザーと、協調的で期限付きのシャットダウン経路を提供します。worker が panic
または予期せず終了した場合は指数バックオフで再起動します。一方、意図した drain では
新しい処理の取得を止め、期限付きのクリーンアップを完了し、代替処理を再起動しません。

シャットダウン時は、タスクを生成する HTTP トラフィックを先に排出してから worker
コンシューマーを停止します。プロセスは最初に readiness を失い、実行中の HTTP リクエストを
完了させ、worker を drain します。必要な場合は強制キャンセルを行い、最後に Redis と MySQL
リソースを閉じます。リスナー障害も `SIGINT` や `SIGTERM` と同じライフサイクルに入り、
`log.Fatalf` で直接終了しません。

本リリースでは、データベーススキーマバージョン、HTTP またはアップロード API、
画像処理レシピを変更しません。

## 変更点

### 統合 worker 監視

- heartbeat、outbox、V2 publish、V2 cleanup、view flush、cleanup、user deletion の
  各 worker を 1 つの名前付きスーパーバイザーで実行します。
- worker の panic を捕捉し、予期しない終了後は 1、2、4、8、16、30 秒という上限付きの
  指数遅延で再起動します。それ以降の失敗も 30 秒を上限とします。
- 障害が発生した worker 名と元の panic 値を記録し、プロセス全体を終了させずに
  繰り返す障害を特定できるようにします。
- `Manager.Start(ctx)` はすべての監視対象 worker が停止するまでブロックし、冪等かつ
  並行安全な `BeginDrain()` シグナルを提供します。
- drain または強制キャンセルの開始時に再起動待機を直ちに中断し、どちらかの停止シグナル後は
  worker を再起動しません。

### プロデューサーを先に、コンシューマーを後に止める順序

- シャットダウン開始直後、`/ready` は HTTP 503 と `status: draining` を返します。
  `/health` の動作は変わりません。
- worker に新しい処理の取得停止を求める前に HTTP サーバーを drain します。これにより、
  実行中のアップロードは永続 publish タスクと outbox レコードを作成し終えてから、
  コンシューマーが最後の drain を実行できます。
- 最初にソフト drain フェーズで worker を待機し、必要な場合は worker のルート context を
  キャンセルして、独立した強制停止フェーズでもう一度待機します。
- worker が完了するか、worker 停止予算が期限切れになった後にだけ Redis と SQL プールを閉じます。
- 予期しないリスナー終了を同じシャットダウンコーディネーターへ渡し、ライフサイクル経路では
  fatal logger を使わずに停止エラーを報告します。

### 期限付きライフサイクル予算

- アプリケーション全体のシャットダウン予算を 30 秒に制限します。内訳は HTTP drain が
  最大 18 秒、worker のソフト drain が 5 秒、強制停止が 5 秒、実行時リソースの終了が
  2 秒です。
- Compose バックエンドの `stop_grace_period` を 40 秒に延長し、30 秒のアプリケーション
  予算とコンテナーランタイムのスケジューリングに余裕を持たせます。
- 最終 view flush とリースクリーンアップに新しい期限付き context を使い、通常の worker
  context がキャンセルされた後でもクリーンアップを実行できるようにします。

### 永続 publish および outbox リース

- drain 開始後、V2 publish と outbox worker は新しい行を取得しません。すでに取得した処理は
  ソフト drain の間に完了できます。
- 中断された publish と outbox の claim を、新しい 3 秒のクリーンアップ context で解放します。
- すべての解放を lease owner と lease token の両方でフェンスします。古い worker の
  クリーンアップは、完了済みの行や代替 worker が取得したリースを解除できません。
- 通常のキャンセルでは claim による試行回数を戻し、シャットダウンだけで再試行予算を
  消費しないようにします。
- publish の panic では試行回数を保持し、panic をスーパーバイザーへ再送出します。
  これにより、決定的な障害は引き続き試行上限へ到達できます。
- outbox 配信の panic では、現在処理中のイベントだけを失敗にし、未処理の取得済みイベントの
  試行回数を戻してから、元の panic をスーパーバイザーへ再送出します。

### 最終 view count flush

- ソフト drain または強制キャンセルのどちらでも、新しい context を使って 1 回の最終
  view count flush を実行し、最大 3 秒に制限します。
- Redis の `GETDEL` 後に MySQL 更新が明示的エラーを返した場合、新しい 1 秒の Redis
  `INCRBY` context で取り出したカウントを復元し、黙って失わないようにします。
- 復元処理を 5 秒の worker 強制停止予算内に収めます。

### 継続的インテグレーション

- 固定バージョンの MySQL 8.4.10 CI サービスに専用のライフサイクルテスト用スキーマを追加します。
- publish リースの owner/token フェンス、publish panic 後のリース解放と panic 伝播、
  outbox リースの owner/token フェンスを実 MySQL で検証する必須テストを追加します。
- スーパーバイザー、再起動バックオフ、drain 順序、ライフサイクル期限、readiness gate、
  view flush、および Compose 猶予期間のユニットテストを追加します。

## インストール

開発イメージは明示的に pull する必要があります。

```bash
docker pull jaykserks/summerain:dev-v2.0.4
```

同等の正確な開発タグ:

```text
jaykserks/summerain:dev-2.0.4
```

移動可能な `dev` タグも最新の成功した開発ビルドを指します。本リリースでは、
`latest`、`main`、安定版セマンティックバージョンのエイリアスは変更されません。

v2.0.3 からのアップグレードにデータベースまたは API の移行は不要で、ローリングアップグレードを
利用できます。ただし本番環境では、移動可能な `dev` タグではなく、正確なイメージタグまたは
digest を固定してください。

## 既知の制限

- view count は、トランザクションまたは冪等な台帳なしで Redis と MySQL をまたぎます。
  MySQL が更新をコミットした後に結果不明のエラーを返すと、Redis へ復元したカウントが
  後の flush でもう一度適用される可能性があります。新しい復元経路は明示的な失敗での損失を
  防ぎますが、ストレージ間の exactly-once 配信を保証しません。
- heartbeat と user-deletion worker はバッチ間で停止しますが、実行中バッチ内のすべての
  データベース操作がまだ完全に context 対応しているわけではありません。実行中の DB 呼び出しは、
  ドライバー処理が戻るか、プロセス停止期限が適用されるまで残る可能性があります。
- 厳密な設定解析と検証は v2.0.5 で予定されています。
- 本スーパーバイザーが対象とするのは、`Manager` が所有するバックグラウンド worker だけです。
  旧 V1 AVIF 変換を含め、リクエスト内で実行されるリソース負荷の高い経路は管理対象外です。
  これらのリソースアドミッションおよび制限制御は v2.0.6 で予定されています。
- 実行時依存関係と worker health の完全な readiness チェックは v2.0.7 で予定されています。
  v2.0.4 では、シャットダウン drain 中に readiness を直ちに失敗させる処理だけを追加します。
- アニメーション画像への対応は今後のリリースで予定されています。
