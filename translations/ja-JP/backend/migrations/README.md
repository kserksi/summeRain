# スキーママイグレーション

バックエンドは `repository.ApplySchemaMigrations(db)` を通じて、追加のみを行う
MySQL スキーママイグレーションを管理します。この関数がエラーを返した場合、起動処理は
直ちに停止しなければなりません。

マイグレーションランナーは次の処理を行います。

- 1 本の SQL 接続を占有し、データベーススコープの MySQL アドバイザリーロックを保持する。
- 必要に応じて `schema_migrations` を作成する。
- 適用済みの各バージョンについて、不変の SHA-256 チェックサムを検証する。
- 不足している操作をバージョン順に適用する。
- 列またはインデックスを追加する前に `information_schema` を確認する。
- そのバージョンのすべての操作が成功した後に限り、バージョンを記録する。

MySQL の DDL は暗黙的にコミットされます。そのため、すべての操作は追加型で、
中断後に安全に再試行できなければなりません。適用済みマイグレーションは決して編集せず、
より大きいバージョンを追加してください。破壊的な名前変更、列削除、テーブル削除には
個別の互換性計画とロールバック計画が必要です。このため、V2 の初期マイグレーションには
意図的に含めていません。

現在のバージョン：

- `2026071501`：`images` に V2 パイプラインのメタデータとインデックスを追加する。
- `2026071502`：V2 バリアント、アップロードセッション/パート、処理ジョブ、
  トランザクショナルアウトボックスの各テーブルを作成する。
- `2026071601`：バックエンドインスタンス間でグローバルアップロード容量の予約を
  直列化するための単一行を追加する。
- `2026071602`：ストレージ参照検索用インデックスと、上限付き制御プレーン保持
  インデックスを追加する。
- `2026071603`：リモートバックエンド、エンドポイント、バケットの系統情報を保持する列とインデックスを
  追加する。履歴行は独立した移行ツールのため未分類のままとし、サーバー起動時に
  ストレージターゲットを推測することはない。
